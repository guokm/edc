# 公司演示剧本（角色分工版，12 分钟）

## 1. 演示目标

按角色讲清楚“谁做什么”：

1. 供应方：发布资产并提供数据
2. 消费方：发现资产、协商签约、发起传输并取数
3. 运营方：治理巡检、按次计费、节点健康监控

## 2. 演示前准备

```bash
./scripts/verify.sh
```

访问前端：`http://localhost:18080`

## 3. 演示路径（按角色切换）

### 3.1 供应方（3 分钟）

- 入口：`供应方入口 -> 供应方工作台`
- 点击：`刷新策略列表`（或`创建默认策略`）-> `发布资产（供应方）`
- 观察：发布轨迹显示“创建资产 -> 写入 Offer -> 刷新目录”
- 话术：供应方只负责“发布与交付”，不负责协商发起

### 3.2 消费方（5 分钟）

- 入口：`消费方入口 -> 消费方工作台`
- 前置条件：运营方已为 `participant-b` 创建/存在 `ACTIVE` 会员（否则协商与传输会被 `403` 拒绝）
- 点击：`刷新目录` -> 选资产 -> `执行签发并签约`
- 点击：协议列表 `用于传输` -> `演示编排门禁（Agreement）`（先展示会员/资格/计费/选 Data Plane）
- 再点击：`发起传输`
- 点击：`拉取数据`
- 观察：签发轨迹（Issuer/Identity/DCP）+ 协商 ID、协议 ID、编排门禁步骤 + 传输状态、链路轨迹、payload
- 失败判定：若未完成签发资格（DCP 校验）即协商/传输，会返回 `403`
- 话术：消费方完成从“发现资产”到“拿到数据”的闭环

### 3.3 运营方（4 分钟）

- 入口：`运营方入口 -> 治理模块接口`
- 点击：`一键巡检`、`额度校验`、`审计列表`、`账单列表`
- 切页：`节点与健康` -> 刷新状态
- 观察：治理接口 200、计费字段、审计事件（ASSET/NEGOTIATION/TRANSFER）、账单记录、双数据平面运行摘要
- 话术：运营方提供可治理、可计费、可观测能力

## 4. 一眼看懂版

- 单页速览：`docs/one-page-demo.md`
- 页面内置：每个角色下均有 `角色演示文档` 页面（3 步脚本 + 验收点）

## 5. 现场常见问答

- **Q：这是不是 Demo 假数据？**
  - A：不是，协商、协议、传输、EDR、计费均持久化到 MySQL。
- **Q：怎么证明职责分离？**
  - A：角色入口隔离了可操作页面，供应方/消费方/运营方操作不同。
- **Q：双数据平面是否参与流程？**
  - A：可在节点健康页与传输状态中看到 `dp-1`/`dp-2` 的实际数据。
